redis对象

redis

对象

redis基于数据结构创建了对象系统，每个对象使用了至少一种数据结构。可以根据不同的场景使用不同的对象实现，优化使用效率

redis对象系统还实现了基于引用计数的内存回收机制，当程序不在使用某个对象，内存就会被释放。

redis通过引用计数实现了对象共享机制，在适当条件下让多个键共享同一个对象节约内存

redi锁对象带有访问时间信息，可以用于计算数据库键的空转时长，空转时长较大的键可能会优先删除

对象类型和编码

redis的每个对象对应一个redisObject结构，包含type、编码和指向实现的指针。

type：字符串对象、R列表对象、H哈希对象、S集合对象、Z有序集合对象。

redis键总是字符串对象。

TYPE key返回key对应的val的类型

OBJECT ENCODING返回key对应的val的编码

  1. 压缩列表比双端链表节约内存，并且元素少时，内存中连续存储，可以更块载入缓存
  2. 随着列表对象元素增多，压缩列表的优适消失，转化为双端链表

字符串对象

字符串对象的编码可以是int、raw或者embstr

如果字符串对象保存整数，并且可以用long类型表示，则保存为int编码

如果是一个字符串值，并且长度大于32字节，使用raw编码，SDS来保存字符串

如果是一个小于32字节的字符串，使用embstr编码保存字符串

embstr是专门保存短字符串的一种优化编码方式，这种编码和raw编码一样，但是raw编码会调用两次内存分配创建redisObject结构和sdshdr结构，embstr通过一次内存分配连续的空间保存字符串。

  1. 内存分配和内存释放都只需要调用一次
  2. 可以更好的利用缓存优适

double等类型也是字符串存储，先将double变成字符串然后存储

编码转化
  1. 对于int编码的字符串对象，如果从整数变成了字符串，编码变为raw
  2. embstr没有任何修改命令，对embstr修改会变为raw再修改

列表对象

列表对象的编码可以是ziplist或者linkedlist

ziplist编码的列表对象使用压缩列表作为底层实现，每个压缩列表节点保存一个列表元素。

linkedlist编码的列表对象使用双端队列作为底层实现。每个节点都保存了一个字符串对象，每个字符串对象保存了一个列表元素。（只有字符串对象会被其他四种类型队列嵌套）

编码转换

当满足以下两个条件时，列表对象使用ziplist编码（可以修改list-max-ziplist-value|list-max-ziplist-entries）

  • 列表所有字符串元素长度小于64字节
  • 列表对象保存的元素数量小于512个

哈希对象

哈希对象的编码可以是ziplist或者hashtable

ziplist编码的哈希对象使用压缩列表作为底层实现，每当有新的键值对加入哈希对象时，先把键压入列表尾部，然后将value压入列表的尾部

hashtable编码的哈希对象使用字典作为底层实现，哈希对象中每个键的值都使用一个字典键值对来保存

编码转化

当满足下面两个条件时，哈希对象使用ziplist编码（可以修改hash-max-ziplist-value|hash-max-ziplist-entries）

  • 哈希对象的键和值长度都小于64字节
  • 哈希对象保存的键值对数量小于512个

集合对象

集合对象的编码可以是intset和hashtable

intset编码的集合对象使用整数集合作为底层实现，集合对象包含的所有元素都被保存再整数集合中

hashtable编码的集合对象使用字典作为底层实现，字典的键是字符串对象，value是null

编码转化

当满足下面两个条件时，使用intset编码（可以i需改set-max-intset-entries）

  • 集合对象保存的都是整数
  • 集合对象保存的元素数量不超过512个

有序集合对象

有序集合的编码是ziplist或者skiplist

ziplist编码的有序集合对象使用压缩列表作为底层实现，每个机会元素使用两个紧挨着的压缩列表节点保存，第一个节点保存元素的成员，第二个节点保存元素的分值。压缩列表元素按分支大小排序。

zet结构中中的dict字典为有序集合创建了从成员到分值的映射，可以O(1)查找分值

skiplist编码的有序集合对象使用跳表作为底层实现

编码转换

当满足下面两个条件，使用ziplist编码（可以修改zset-max-ziplist-value|zet-max-ziplist-entries）

  • 有序集合所有元素长度小于64字节
  • 有序集合保存的元素数量小于128

命令检测与命令多态

redis操作键的命令基本分为两种，可以对任何键使用的命令和对特定键使用的命令

类型检查实现

类型特定命令的检查通过redisObject的type实现：命令执行前，数据库检查键对应的值对象是不是执行命令的类型，是就执行，否则返回错误类型

多态命令的实现

redis还会根据对象的编码方式判断命令，一个redisObject可能有多种底层实现，需要执行时判断使用哪种实现的api

内存回收

c语言没有内存回收功能，redis在对象系统构建了一个引用计数refcount，当计数变为0时， 对象所占内存会被释放

对象共享

对象的引用计数还带有对象共享的作用，redis初始化时，会创建一万个字符串对象，0-0000，当需要用到这些值时，就使用共享对象

对象空转时长

redisObject还有一个lru属性，记录对象最后一次被访问的时间。OBJECT IDLETIME msg查看。

当服务器打开maxmemory选项，并且内存回收算法为lru，redis会优先释放空转时间较长的 键

redis数据结构

redis

数据结构

简单动态字符串

redis没有使用c语言的字符串表示，自己构建了一种简单动态字符串类型SDS。

c的字符串只用在一些无须对字符串值修改的对方

SDS处理保存字符串值，还用作缓冲区：AOF缓冲区、客户端状态的输入缓冲区等

SDS定义

末尾加’\0’符合c的结构，可以使用c的api

区别

C字符串不保存长度，获取长度需要遍历字符串，SDS直接访问len即可

C字符串如果没有提前分配空间，会产生缓冲区溢出，覆盖相邻位置的值。SDS进行修改时，会检查空间是否足够，不够就会扩容。

C每次对字符串修改，都会重新分配内存，否则会内存溢出或内存泄漏。

由于redis经常对SDS修改，所以不能频繁进行内存分配，SDS使用了空间预分配和懒惰空间释放

空间预分配

当对SDS修改时并需要扩展时，不仅分配需要的空间，还额外分配未使用的空间

  • 如果修改后的值小于1MB，free大小和len大小相同
  • 如果大于1MB，则free空间为1Mb
惰性空间释放

当SDS缩短时，不会立即进行内存回收，使用free记录这些字节的数量，等待将来使用

二进制安全

C的字符串必须符合某种编码，除了末尾其他地方不能由空字符串，否则认为字符串结束。限制了C字符串保存的数据类型。

SDS不会对数据进行任何限制，直接保存，并且不会有二进制安全问题。

链表

redis构建了自己的链表实现。列表键的底层实现之一就是链表

包含链表和迭代器，是一个双向队列

迭代器包含内存指针，0代表正方向迭代，从头开始，1代表反方向迭代，从尾开始

可以传入的自定义dup、match、free等方法进行链表的复制、匹配、释放等

字典

字典是数组里存放了一个ditEntry，每个entry包含key和value对象类型，value链表的指针

字典包含了：

  • 字典类型
  • 两个hash表，为了rehash，扩容时使用，一般放在h0
  • rehashidx，rehash标识，是否正在rehash，-1不能rehash 当数量/容量 >5 强制rehash

hash表：

  • entry：指向指针集合，可以理解为数组
  • hash表大小
  • 计算掩码（size-1）等同于java
  • use，已使用大小

迭代器：

  • d：指向的字典
  • index：索引的位置
  • safe：标识迭代器是否安全，某些操作会被禁止
  • entry指针：当前和下一个两个指针
  • fingerprint：hash值，判断hash表是否修改

逻辑

dictResize：把hashTable变成最小的包含所有元素的表，缩小hashTable

  1. 判断是否可以reHash或是否正在reHash
  2. 当使用/总bucket小于1时，进行缩小，最小是4
  3. 计算新的hashTable的大小
  4. 进行扩张
    1. 初始化新的hashTable
    2. 如果已经有0号hashTable，就把新的hashTable付给1号，然后标记可以rehash，从0到1开始hash

rehash：将0号复制到1号。

无论扩张还是收缩，都是变成大于used大小的下一个2^n

条件：

  1. 服务器没有执行BGSAVE或BGREWRITEAOF，并且负载因子大于1
  2. 服务器正在执行BGSAVE或BGREWRITEAOF，并且负载因子大于5
  3. 负载因子小于0.1，进行收缩

定时任务，每个确定的ms内进行rehash，搬运100个，server启动时注册到了时间函数中

  1. 循环所有bucket
  2. 对每个链表的每个节点进行rehash并放入h1
  3. 修改h0和h1的used数值
  4. h0和h1交换

dictAddRaw：添加新的键值

  1. 查找是否相同的key，如果有返回-1
  2. 如果正在rehash，就确定要添加到h1，否则h0
  3. 分配entry空间，头插法插入entry
  4. 将entry填入key和val，如果已存在key，直接覆盖

dictGenericDelete：删除

  1. 对两个表进行删除
  2. 对每个表循环，找到key相同的entry，并删除节点
  3. 修改use值即可

find和delete相同

dicNext：迭代器

  1. 初次迭代时初始化迭代器类型（安全还是不安全）
  2. 如果迭代到表后，判断是否rehash，如果是，取h1表迭代
  3. 如果entry为null，取下一个桶取，否则取链表的下一个entry

redis跳表

结构

zskiplist：跳表

  • header：头指针
  • tail：尾指针
  • length：长度
  • level：当前最高的层

zskiplistNode：跳表节点

  • sds：字符串值
  • score：分值，排序zskiplistNode，可以相同
  • backword：zskiplistNode指针
  • zkiplistLevel[]：level数组，层高随机
    • forward：zskiplistNode指针，指向下一个同层
    • span：到一个节点中间跳过了几个节点，没跳就是1

跳表的对象必须唯一，但是分值可以相同，按对象的字典序排序

逻辑

zslCreate：创建zskipList

  1. 初始化空间，level=1，初始化头节点，level是32

zslRandmomLevel：随机产生层数，最大是32层

zslInsert：插入节点，传入string和分数

  1. 从头节点的最高处开始查找，每层的rank表示上一层跳表的个数
  2. 在每一次一直找到大于score或者score和string都相同的节点时退出
  3. 创建新的节点，如果高度高于当前跳表高度，跳表头进行链接
  4. 插入节点，修改跳表并计算跳表距离
  5. 设置后退指针

整数集合

整数集合(intset)是集合键的底层实现之一，当一个集合只包含整数元素，并且集合元素数量不多时，使用整数集合作为集合键的底层实现

整数集合的redis用于保存整数值的集合抽象数据结构，可以保存int16_t，int32_t，int64_t的整数值

使用contents数组保存元素，且不包含整数项。

升级

当向整数集合添加新元素并且新元素类型长度比现有类型长度长时，需要先进行升级，重新分配数组空间，全部变成新元素类型在保存。

整数数组不支持降级。

好处：

  1. 提升灵活性，可以随便放入任何类型的整数，不必担心类型错误
  2. 节约内存，如果只有小类型整数，不必开大类型整数数组存储

压缩列表

压缩列表是列表键和哈希键的底层实现之一。当列表键只有少量列表项，并且列表项是小整数值或者短字符串，就用压缩列表实现列表键

构成

压缩列表是redis为了节约内存开发的，有一系列特殊编码的连续内存块组成的顺序数据结构。

一个压缩列表包含任意多个节点，每个节点可以保存一个字节数组或一个整数值。

分别是：列表字节数、列表尾部偏移、节点数量、节点、尾部标识

每个节点记录了前一个节点长度（用于从后往前遍历）、当前节点的数据类型和长度、数据内容三部分

连锁更新

由于前面节点的长度发送变化导致的连续多次空间扩展（当前节点无法记录前面节点的长度，需要扩展）成为连锁更新。

  1. 压缩列表需要多个连续的长度介于250-253的节点，才会可能触发连续更新，概率较低
  2. 只有连续更新街道办剁，性能不会影响很大

redis慢查询日志

redis

redis慢查询日志

redis慢查询日志用于记录执行时间查过给定时长的命令请求。

  • slowlog-log-slower-than 选项指定执行时间超过多少微秒的命令会被记录再日志上
  • slowlog-max-len指定服务器最多保存多少条慢查询日志

服务器采用先进先出的方式保存慢查询日志

SLOWLOG GET获取慢查询日志

慢查询记录的保存

服务器状态包含了几个和慢查询日志功能相关的属性：

  • slowlog_entry_id：下一个慢查询日志的id
  • slowlog：慢查询日志链表,每个节点保存一个slowlogEntry结构，包含唯一标识，开始时间，执行时间和命令
  • slowlog_log_slower_than
  • slowlog_max_len

慢查询日志的查询和删除

SLOWLOG GET number：打印指定数量的慢查询日志，没有number默认全部（逆序打印，新的日志在头部）

SLOWLOG LEN：查询慢查询日志长度

SLOWLOG_RESET：清空所有慢查询日志

redis位图

redis

redis位图

redis提供了SETBIT、GETBIT、BITCOUNT、BITOP四个命令处理二进制位数组

SETBIT bit 3 1位数组指定偏移量上的二进制位设置值

GETBIT bit 3获取位数组指定偏移量上的二进制位的值

BITCOUNT bit 统计位数组里面1的个数

BITOP AND | OR | XOR | NOT对多个位数组按位与、或、异或、取反

位数组表示

redis使用字符串对象表示位数组，因为字符串对象使用的SDS结构是二进制安全的。

字符串数组每个字符由八个字节组成（这里是为了看的清晰展示了出来，实际没有） 字节逆序构成字符

GETBIT命令实现

GETBIT bitarr offset返回指定偏移的二进制位值

  1. byte = offset / 8 ，计算偏移是再哪个字节
  2. bit = （offset % 8) + 1，bit是字节的第几个二进制位
  3. 返回对应的二进制值

SETBIT命令

SETBIT bitarr offset bitval将指定的offset位修改位bitval

  1. 计算len = offset / 8 + 1，计算需要的字节长度
  2. 检查SDS长度是否小于len，是就扩容，新位置都为0（多分配两个未使用字节）
  3. byte = offset / 8，找到对应的byte
  4. bit = (offset % 8) + 1，计算二进制位
  5. 修改值并返回oldval

如果是顺序存放bit，则新添加的bit是高位，添加后需要重新排序到前面，增加了复杂性

BITCOUNT实现

简单的遍历效果很差，100MB的数据可能需要遍历8亿次

查表算法
  1. 对有限集合，元素排列方式有限
  2. 对有限长度的位数组，表示的二进制位排列有限

将可能的8位/16位二进制值对应的1的个数存再表中，然后每次取8位/16位进行查表

缺点：

  1. 空间换时间，16位就需要几百KB，再大就无法接收
  2. 查表算法受cpu缓存限制：固定大小的cpu缓存，表越大，缓存不命中的情况越高，影响查表效率
variable-precision SWAR算法

统计位数组非0个个数被称为“计算汉明重量”

swar函数每次可以计算32个二进制位的汉明重量，无须查表那样的额外缓存

redis实现

redis使用了查表和swar两种算法

  1. 查找算法使用键长为8的表，记录了00000000-11111111的所有二进制位的汉明重量
  2. bitcount每次循环载入128个二进制位，调用4次32位的swar计算汉明重量
  3. 使用bitcount时，如果二进制位数量大于等于128，使用swar算法计算
  4. 如果小于128，使用查表法计算

BITOP命令实现

  • BITOP AND 使用&计算所有输入的二进制位的逻辑与结果保存再指定的键上
  • BITOP OR 使用|计算所有输入的二进制位的逻辑或结果保存再指定的键上
  • BITOP XOR 使用^计算所有输入的二进制位的逻辑异或结果保存再指定的键上
  • BITOP NOT 使用~计算所有输入的二进制位的逻辑非结果保存再指定的键上

BITOP AND result x y 将xy与的结果保存再result

  1. 创建空白位数组
  2. 一次对每一个字节进行或运算并保存到空白位数组
  3. 结算结果保存再result上面

redis排序

redis

redis排序

redis的SORT命令可以对列表的键，集合键或者有序集合的键的值进行排序

SORT命令为每个排序的键都创建一个和键长度相同的数组，数组的每个项都是一个redisSortObject结构，根据使用的SORT命令不同，对象结构也不同

SORT key

这个命令对一个包含数字值的key进行排序

  1. 创建一个和列表长度相同的数组，每个项都是一个redisSortObject
  2. 遍历数组，将各个数组的obj指针指向numbers列表的各个项
  3. 遍历数组，将obj指针指向的列表项转化为一个double数，保存至数组项的u.score属性值
  4. 根据u.score的值，对数组的进行数值大小的排序
  5. 遍历数组，将数组的obj指向的列表项返回客户端

ALPHA选项的实现

通过ALPHA选项，SORT命令可以对包含字符串值的键排序

  1. 创建redisSortObject数组
  2. 遍历数组，保存obj指针
  3. 根据obj指向的字符串，对数组进行字符串排序
  4. 返回结果

ASC和DESC

根据选项的不同，排序时是否进行降序排序

BY选项的实现

默认情况，SORT使用排序键的值元素进行排序，BY可以指定某些字符串键或某个哈希键锁包含的某些域进行排序

SORT fruits BY &-price

  1. 创建数组
  2. 指针指向，返回域的值
  3. 保存到u.score属性上
  4. 排序并返回

LIMIT实现

通过limit选项，只返回其中一部分排序号的元素

SORT alphabet furit limit 1 4

1表示跳过的元素数量，4表示返回的数量

GET实现

通过GET选项，让SORT命令在对键排序后，根据被排序的元素和GET指定的模式返回某些键的值

SORT studemts ALPHA GET *-name

返回符合*-name模式的排序后的值

STORE实现

通过STORE选项，将排序结果保存在指定的键里面，并在需要时重用这个排序结果

SORT students STORE sorted_students

STORE会检测sorted_students是否存在，存在就删除该键，然后再保存

多个选项的指向顺序

  1. 排序：命令使用ALPHA、ASC、DESC和BY对输入进行排序
  2. 限制排序长度：这一步会使用LIMIT，对结果集长度限制
  3. 获取外部键：这一步会调用GET，根据排序结果集中的元素和GET模式查找指定的键值作为新的结果集
  4. 保存结果集：使用STORE将结果集保存
  5. 返回排序结果

注意：如果由多个GET命令，要注意执行顺序，不同的执行顺序结果可能不同

redis事务

redis

redis事务

redis通过MULTI、EXEC、WATCH等命令实现事务功能。

事务提供了一种将多个命令打包，一次性按顺序执行多个命令的机制。事务执行时，不会中断事务去执行其他客户端命令。

事务受限以一个MULTI命令开始，最后由EXEC命令将事务commit到服务器执行

事务的实现

事务通常经历三个阶段：事务开始—事务入队—事务执行

事务开始

MULTI命令标志事务开始，MULTI将客户端从非事务状态转至事务状态，通过修改客户端的flags属性中打开REDIS_MULTI标识完成

命令入队

当一个客户端处于非事务状态时，服务器会根据客户端发送的不同命令执行不同的操作

  • 如果是EXEC、DISCARD、WATCH、MULTI命令的其中一个，服务器立即执行
  • 如果是其他命令，服务器不会立即执行，将命令放入事务队列中，向客户端返回QUEUED回复
事务队列

每个redis客户端都有自己的事务状态，这个事务状态保存在客户端状态的multiState对象中，multiState维护了一个事务队列和入队命令的计数器，事务队列是一个multiCmd数组，每个数组保存了一个入队命令相关的信息，包括函数指针，命令的参数，参数数量等

执行事务

当一个处于事务状态的客户端向服务器发送EXEC命令，服务器立即执行，遍历事务队列，执行所有命令，最后将所得结果返回给客户端。

WATCH命令实现

WATCH命令是一个乐观锁，在EXEC命令执行前，监视任意数量的数据库键，在EXEC执行时，检测键是否至少一个被修改，如果是，服务器拒绝执行事务，并返回失败的空回复

WATCH命令监视数据库键

每个redisDB数据库保存一个watched_keys字典，key是被某个WATCH命令监视的数据库键，val是一个链表，记录了所有监视数据库键的客户端

监视机制的触发

对所有数据库进行修改的命令，执行后会调用函数对watched_keys检查，如果有监视刚刚被命令修改过的数据库键，打开链表对应客户端的redis_dirty_cas标识，标识事务安全性被破坏

判断事务是否安全

服务器接收到EXEC命令后，检测客户端是否开启了redis_dirty_cas标识，判断是否执行事务

事务的ACID性质

原子性

redis的事务，要不就全部执行，要不就全不执行。

redis不支持回滚，即使某条命令出错，事务也会继续执行完毕。

一致性

redis通过错误检测和简单的设计保证事务的一致性。redis有3个可能出错的对方

  1. 入队错误：如果事务在入队过程中，出现了命令不存在，命令格式不正确等情况，redis拒绝这个事务
  2. 执行错误：服务器不会中断事务执行，会继续执行剩下的命令，已执行的命令也不会受影响
  3. 服务器停机：如果服务器在执行事务时停机，根据持久化模式，可能有以下情况出现，
    1. 无持久化时，重启后服务器数据库空白
    2. RDB模式下，事务停机导致的不一致，可以通过RDB文件恢复。
    3. AOF模式下，事务停机导致的不一致，可以通过AOF文件恢复。
隔离性

redis使用单线程的方式执行事务，并且事务不会中断，所以具有隔离性

持久性

redis没有提供额外的持久化功能，redis的持久性由持久化模式决定

  1. 当RDB持久化时，由于特定条件才会持久化，所以无法保证持久性
  2. AOF模型下，当appendfsync为always时，具有持久性

redis发布订阅

redis

redis发布订阅

redis的发布和订阅功能由PUBLISH、SUBSCRIBE、PSUBSCRIBE等组成

通过SUBSCRIBE，客户端可以订阅一个或多个频道，成为这些频道的订阅者，每当其他客户端向频道发送下消息时，频道所有订阅者接收到该消息、与这个频道相匹配的模式的订阅者也可以收到该消息

频道的订阅和退订

当一个客户端执行SUBSCRIBE订阅某个频道，这个客户端和频道建立了订阅关系

redis将所有的订阅关系保存在redisServer的pubsub_channels字典里，key是订阅的频道，val是链表，记录了所有订阅该频道的客户端

订阅
  1. 如果频道有其他订阅者，将客户端直接添加到频道订阅者尾部
  2. 如果没有任何订阅者，创建一个key，并将客户端添加到链表，成为链表第一个元素
退订
  1. 如果频道还有订阅者，删除当前客户端信息
  2. 如何频道没有任何订阅者，删除频道

模式的订阅和退订

模式的所有订阅关系都保存在pubsub_patterns里面，pubsub_pattern是一个链表，每个pattern结构记录了被订阅的模式和订阅模式的客户端（1对1）

订阅模式
  1. 新建一个pubsubPattern结构，结构是pattern属性设则为订阅的模式，client设置为订阅的客户端
  2. 将pubsubPattern结构添加到链表尾部
退订模式

查找对应模式的pubsubPattern对象并从链表删除

发送消息

redis客户端发送PUBLISH channel message时

  1. 将message发送给channel频道的所有订阅者
  2. 如果一个或多个pattern与channel匹配，将消息发送给pattern模式的订阅者

查看订阅消息

客户端通过PUBSUB命令查看频道或模式的相关信息

  • PUBSUB CHANNELS pattern返回当前被订阅的频道，如果不写pattern参数，返回所有被订阅频道，如果给定pattern参数，返回与pattern匹配的频道
  • PUBSUB NUMSUB channel-1 channel-2...返回多个频道订阅者的数量
  • PUBSUB NUMPAT返回服务器当前被订阅模式的数量

redis集群

redis

redis集群

redis集群是redis通过的分布式数据库方案，通过分片来进行数据共享，并提供复制和故障转移

节点

一个redis集群有多个节点组成，开始时每个节点相互独立，处于自己的集群。组建集群时，将各个独立的节点连起来。

CLUSTER MEET ip port向一个ip的节点发送握手handshake，握手成功，就将ip节点添加到当前集群

启动节点

一个节点就是一个集群模式下的redis服务器，服务器启动时根据cluster-enabled是否yes判断是否开启集群模式

集群数据结构

每个节点保存着一个clusterState结构，记录了当前节点视角下集群的状态，其中的dict字典记录了集群节点名单

每个节点对应一个clusterNode结构，记录了节点的状态，其中的clusterLink记录了连接节点的有关信息，包括socket描述符，输入输出缓冲区等。

CLUSTER MEET

向节点A发送CLUSTER MEET,客户端让节点A将另一个节点B添加到A的集群

  1. A为B创建clusterNode结构，添加到dict字典
  2. A向ip的B发送MEET消息
  3. B为A创建clusterNode结构，添加到dict字典
  4. B向A返回一条PONG消息
  5. A向B发送一个PING消息，完成三次握手

槽指派

redis通过分片保存数据库的键值对，集群的数据库被分为16384个槽，每个键属于其他的一个槽，每个节点可以处理0-16384个槽

如果16384个槽都在处理，集群处于上线状态，如果有一个没有得到处理，集群处于下线状态

CLUSTER ADDSLOTS 可以将一个或多个槽指派给某个节点

记录节点的槽指派信息

clusterNode的slots属性和numslot记录了节点复制处理的槽

slots是一个二进制位数组，长度位16384/8=2048个字节 numslot记录了槽位1的个数

传播节点的槽指派信息

节点记录了槽指派信息后，会向集群的节点发送自己的slots信息，收到的节点更新对应节点的slots数组

记录集群的槽指派信息

clusterState的slots数组记录的是每个槽对应的节点

为什么两处保存：

  1. 如果只使用clusterNode的slots，想知道某个槽是否指派时，需要遍历所有节点
  2. 程序需要将某个节点的槽指派信息发送时，直接发送clusterNode的slots即可

clusterState的slots数组记录了所有槽的指派信息，ClusterNode的slots记录了节点的槽指派信息

集群中执行命令

集群的槽都被指派后，集群上线

客户端命令的处理流程：

客户端接收到MOVED错误后，会转型到对应的节点，重新发送命令

计算属于哪个槽

CRC16(key) & 16383

计算槽后，判断ClusterState的slots[i]是不是当前节点，是就处理，不是就将对应节点的ip和port发送给客户端

节点给客户端发送MOVED slot ip port

集群客户端不会打印MOVED错误，会根据MOVED信息进行转向，打印转向信息

单机模式会报MOVED错误，因为单机模式无法处理MOVED错误

节点数据库的实现

节点数据库使用和单机数据库使用一样，但是节点数据库只能使用0号数据库

节点的clusterState的slots_to_keys跳表记录了每个槽对应的key，得分socre是槽号

通过slots_to_keys记录各个数据库键所属的槽，节点可以很方把的对属于某个或某些槽的键批量操作

重新分片

redis重新分片可以把任意数量的槽从某个节点指派到另一个节点，槽的键值对也会被移到对应的节点

重新分配可以在上线期间操作。

重新分片由redis集群管理软件redis-trib执行

  1. redsi-trib对目标节点发送importing命令，让目标节点准备号从源节点导入属于slot的键值对
  2. redis-tirb对源节点发送migrating命令，让源节点准备好将属于slot的键值对迁移到目标节点
  3. redis-trib对源节点发送getkeysinslot，获得最多count个key
  4. 对每个key，redis-trib向源节点发送migrate命令，将该键值对迁移到目标节点
  5. 当全部迁移完成，redis-trib向集群任意节点发送setslot命令，将槽指派给目标节点，这一指派信息会发送到整个集群

importing_slot_from数组表示当前节点导入对应的节点的槽i

migrating_slot_to数组表示当前节点将槽i迁移到对应的节点

当迁移期间客户端请求数据，源服务器如果找到数据就返回，找不到就返回一个ASK错误，指引客户端向目标服务器发起请求查找数据

同MOVED错误

ASK和MOVED区别：

MOVED错误代表槽的负责权已经从一个节点转移到另一个节点，之后客户端就会把该槽的命令发送到新节点

ASK错误是节点迁移过程中的一种临时措施，不会转移客户端的命令，只是临时下一次命令发送到新节点

负责和故障转移

redis集群分为主从节点，主节点处理槽，从节点复制主节点数据，主节点下线时代替主节点

设置从节点

CLUSTER REPLICATE node_id让接收命令的节点变成node_id节点的从节点，并开始进行复制

  1. 从节点会将自己clusterNode的从节点指针指向主节点
  2. 从节点修改clusterState的自我标识，变为从节点标识
  3. 调用复制代码，根据主节点指针的ip和端口号进行复制
故障检测

集群的每个节点都会定期向集群其他节点发送ping命令。检测对方是否在线，如果没有在规定时间返回pong命令，就会将该节点标记为疑似下线

如果集群内半数以上的主节点都将某个节点标记疑似下线，该主节点标记为已下线，标记节点下线的节点会发送消息，通知该节点下线

故障转移

从节点发现主节点下线，开始执行故障转移

  1. 复制下线节点的所有从节点里，会有一个节点被选中
  2. 选择节点执行slave no one命令，成为主节点
  3. 主节点撤销所有已下线节点的槽指派，全部指派给自己
  4. 广播一条pong消息，通知该节点变成了主节点，并且已经接管了槽
选举新节点
  1. 集群的配置纪元是自增计数器，初始为0
  2. 每进行一次故障转移，进去配置纪元+1
  3. 每个配置纪元，集群中主节点有一次投票机会，第一个要求投票的从节点获得投票
  4. 从节点广播投票，主节点返回支持
  5. 超过半数的投票节点成为主节点，如果没有节点收集到半数票，配置纪元+1，重新选举

消息

集群中节点发送和接收消息来通信。

消息主要是5中：

  1. MEET消息：接收者加入到发送者的集群
  2. PING消息，每个节点默认1秒从所有节点选取5个节点， 然后选择最长时间没发ping的节点发送ping命令，除此之外，如果节点返回pong消息的时间到当前时刻超过预设的timeout的一半，也会发送ping命令
  3. PONG消息：当接受者接收到MEET、PING消息时，发送PONG消息确认接收。节点可以向集群广播PONG消息让集群更新节点的认识
  4. FAIL消息：节点A判断节点B下线，广播一条B的FAIL消息，标记B下线
  5. PUBLISH消息：节点接收到PUBLISH消息，执行这个命令，并向集群广播一条PUBLISH消息，所有接收到该命令的节点也执行PUBLISH消息

一条消息由消息头和正文组成
